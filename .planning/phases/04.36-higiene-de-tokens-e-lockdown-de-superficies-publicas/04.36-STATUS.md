# 04.36 Status

## Contexto

Fase P0 aberta para fechar rapidamente riscos de segredos em URL, exposicao desnecessaria em readiness e dependencia fraca de rate limiting antes de ampliar qualquer uso do piloto.

## Implementado

### Websocket sem token em query string

- O frontend operacional deixou de montar `?token=...` na URL do websocket.
- A autenticacao agora acontece no primeiro frame do socket:
  - cliente abre `ws(s)://.../ws/updates`
  - cliente envia `{"type":"auth","token":"..."}`
  - backend valida antes de registrar a conexao no `websocket_manager`
- O estado de tempo real agora so fica `conectado` depois do evento `connected`.

Arquivos:

- `saas-frontend/src/pages/dashboard/OperationalDashboardPage.tsx`
- `saas-backend/app/main.py`

### Reset de senha sem token em query string

- O backend agora gera links no formato:
  - `/reset-password#token=...`
- A SPA ganhou rota publica dedicada para reset:
  - `/reset-password`
- A nova tela preenche o token a partir do fragmento da URL.
- Como fallback de compatibilidade, a tela ainda consegue ler `?token=` localmente caso exista algum link antigo, mas o backend nao emite mais esse formato.

Arquivos:

- `saas-backend/app/services/auth_service.py`
- `saas-frontend/src/pages/auth/ResetPasswordPage.tsx`
- `saas-frontend/src/App.tsx`

### Webhook publico sem token em query string

- O webhook publico do WhatsApp deixou de aceitar token pela query string.
- Agora ele aceita apenas:
  - `X-Webhook-Token`
  - `Authorization: Bearer ...`

Arquivo:

- `saas-backend/app/routers/public.py`

### `/health/ready` endurecido

- O readiness continua util para deploy e probe.
- Ele nao devolve mais erro bruto de banco/cache no payload publico.
- O payload agora expoe apenas status de alto nivel.

Arquivo:

- `saas-backend/app/main.py`

### Startup falha em producao sem rate limiting real

- O backend agora trava startup se `environment=production` e o rate limiting real nao estiver habilitado.
- Isso remove o fallback permissivo como opcao silenciosa em producao.

Arquivo:

- `saas-backend/app/core/limiter.py`

### Build de frontend falha em producao sem API base

- O frontend ganhou `runtimeConfig` centralizado.
- Em build de producao, a ausencia de `VITE_API_BASE_URL` agora derruba a build em vez de cair silenciosamente para `127.0.0.1`.

Arquivos:

- `saas-frontend/src/services/runtimeConfig.ts`
- `saas-frontend/src/services/api.ts`

## Hardening adicional em 2026-04-09

- a validacao de `Origin/Referer` para `refresh` e `logout` ganhou regressao explicita cobrindo fallback por `Referer` e reject fora da allowlist.
- toda a superficie `/api/v1/auth/*` passou a responder com `Cache-Control: no-store`, `Pragma: no-cache` e `Expires: 0`, inclusive em erros de origem nao autorizada.
- `frontend_url` e `cors_origins` passaram a ser normalizados para origem canonica, evitando mismatch por barra final/path em producao.
- a configuracao de producao agora falha cedo se:
  - `CORS_ORIGINS` usar `*`
  - `FRONTEND_URL` nao estiver presente em `CORS_ORIGINS`
- o baseline de headers HTTP ganhou `X-Permitted-Cross-Domain-Policies: none` e CSP com `object-src 'none'`.
