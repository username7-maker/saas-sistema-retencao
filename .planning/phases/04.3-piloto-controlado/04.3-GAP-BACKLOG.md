# Phase 4.3 Gap Backlog

## Objetivo

Transformar os gaps identificados no piloto em backlog tecnico priorizado, com foco em:

- reduzir promessa maior que a entrega real
- fechar integracoes externas criticas
- explicitar onde hoje existe fallback/manual
- preparar o produto para sair de `core funcionando` para `operacao mais completa`

## Leitura Geral

O core do piloto esta funcional, mas ainda existem recursos em quatro grupos:

1. fluxo existe, mas esta desligado por configuracao
2. fluxo existe, mas depende de integracao externa nao concluida
3. fluxo existe, mas fecha em fallback/manual
4. superficie visual existe, mas ainda nao deve ser vendida como capacidade plena

## P0 - Corrigir Agora

### P0.1 - Fechar WhatsApp no piloto

- Status atual:
  - API existe
  - frontend exibe estados relacionados a WhatsApp
  - no ambiente sem configuracao o backend responde `Evolution API nao configurada`
- Arquivos principais:
  - `saas-backend/app/routers/whatsapp.py`
  - `saas-backend/app/services/evolution_service.py`
  - `saas-frontend/src/pages/dashboard/RetentionDashboardPage.tsx`
  - `saas-frontend/src/components/tasks/TaskDetailDrawer.tsx`
  - `saas-frontend/src/components/tasks/TasksOnboardingTab.tsx`
- Impacto:
  - automacoes e contatos operacionais perdem uma das alavancas mais importantes do produto
  - a UI mostra capacidade de WhatsApp, mas o piloto ainda nao entrega isso de verdade
- O que falta:
  - configurar `WHATSAPP_API_URL`
  - configurar `WHATSAPP_API_TOKEN`
  - configurar `WHATSAPP_INSTANCE`
  - configurar `WHATSAPP_WEBHOOK_TOKEN`
  - validar QR, status, webhook e mensagem real
- Definicao de pronto:
  - owner conecta instancia
  - status muda para `connected`
  - webhook entra no backend
  - ao menos um fluxo operacional real usa WhatsApp com sucesso

## P3 - Evolucao De Produto

### P3.1 - Telemetria de disponibilidade por feature

- Objetivo:
  - cada modulo “sensivel” mostrar claramente se esta:
    - ativo
    - parcial
    - fallback
    - bloqueado por configuracao
- Beneficio:
  - reduz duvida de operacao
  - reduz suporte manual

### P3.2 - Matriz de readiness por integracao

- Objetivo:
  - painel interno para owner/manager com readiness de:
    - email
    - WhatsApp
    - Actuar
    - IA
    - endpoints publicos
- Beneficio:
  - evita descobrir bloqueio somente quando o usuario tenta usar

## Recomendacao Pratica

Se a meta for dar mais seguranca ao piloto sem abrir muita frente nova, a ordem mais inteligente e:

1. WhatsApp
2. SendGrid
3. decisao explicita sobre bioimpedancia assistida no piloto
4. decisao explicita sobre Actuar no piloto
5. transparencia de fallback de IA
6. endpoints publicos

## Leitura Final

O produto hoje nao sofre de falta de core. O que ainda esta “faltando” de forma perceptivel e a camada de:

- canais
- integracoes
- IA real
- fluxos publicos

Esses pontos precisam ser tratados menos como “bug solto” e mais como backlog de readiness do produto.
