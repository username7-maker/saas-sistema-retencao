# Phase 4.3 Deploy Plan

## Contexto GSD

Este plano foi fechado usando a trilha do GSD para a fase atual:

- `gsd-progress`: fase atual confirmada como `4.3`
- `gsd-discuss-phase --auto`: defaults travados do deploy do piloto
- `gsd-plan-phase`: traducao desses defaults em um plano executavel

## Objetivo

Colocar o piloto controlado no ar com o menor risco possivel, sem ampliar escopo de produto e sem depender de integracoes opcionais para provar o core operacional.

## Defaults Fechados

- frontend: `Vercel`
- API: `Railway`
- worker: `Railway`
- Redis: `Railway`
- Postgres: `Supabase`
- rollout: `manual no primeiro deploy`
- escopo funcional do piloto:
  - importacao
  - members
  - CRM
  - tasks
  - retencao
  - assessments / profile 360
  - users
- desligado no go-live:
  - Claude
  - leitura assistida de bioimpedancia
  - Actuar

## Workstreams

### Workstream 1 — Infra Basica

1. Criar projeto no Supabase.
2. Criar projeto no Railway com:
   - `api`
   - `worker`
   - `redis`
3. Criar projeto na Vercel apontando para `saas-frontend`.
4. Preencher os templates:
   - `deploy/railway-api.env.example`
   - `deploy/railway-worker.env.example`
   - `deploy/vercel.env.example`

### Workstream 2 — Configuracao Correta Do Runtime

#### API

- start command: `uvicorn app.main:app --host 0.0.0.0 --port $PORT`
- flags obrigatorias:
  - `ENABLE_SCHEDULER=false`
  - `ENABLE_SCHEDULER_IN_API=false`

#### Worker

- start command: `python -m app.worker`
- flags obrigatorias:
  - `ENABLE_SCHEDULER=true`
  - `ENABLE_SCHEDULER_IN_API=false`

#### Ambos

- `SCHEDULER_CRITICAL_LOCK_FAIL_OPEN=false`
- `PUBLIC_OBJECTION_RESPONSE_ENABLED=false`
- `PUBLIC_PROPOSAL_ENABLED=false`
- `ACTUAR_ENABLED=false`
- `ACTUAR_SYNC_ENABLED=false`
- `ACTUAR_SYNC_MODE=disabled`
- `BODY_COMPOSITION_IMAGE_AI_ENABLED=false`

### Workstream 3 — Banco E Migrations

1. Ajustar `DATABASE_URL` no formato SQLAlchemy:

```text
postgresql+psycopg2://USER:PASSWORD@HOST:PORT/DBNAME
```

2. Rodar:

```bash
cd saas-backend
set DATABASE_URL=postgresql+psycopg2://USER:PASSWORD@HOST:PORT/DBNAME
alembic upgrade head
```

3. Validar que o banco ficou em `head` antes de liberar o frontend.

### Workstream 4 — Deploy Manual Do Release De Piloto

1. Publicar a API no Railway.
2. Validar `GET /health/ready = 200`.
3. Publicar o worker no Railway.
4. Confirmar logs de jobs e locks.
5. Publicar o frontend na Vercel.
6. Atualizar:
   - `FRONTEND_URL`
   - `CORS_ORIGINS`
   - `PUBLIC_BACKEND_URL`

### Workstream 5 — Bootstrap Da Academia Piloto

1. Criar owner inicial via `POST /api/v1/auth/register`.
2. Confirmar login com:
   - `gym_slug`
   - `email`
   - `password`
3. Criar os usuarios reais:
   - `owner`
   - `manager`
   - `receptionist`
   - `trainer`

### Workstream 6 — Smoke Test Obrigatorio

1. Login e refresh token.
2. Criacao de membro manual.
3. Importacao pequena com `preview -> commit`.
4. Retencao, tasks, assessments e profile 360 sem erro.
5. Recalc manual de risco:
   - `POST /members/recalculate-risk`
   - acompanhar `request_id`
6. CRM de recepcao em modo leitura.
7. Task-lite do trainer no fluxo tecnico.
8. WebSocket em `/ws/updates`.
9. Email real.
10. Um fluxo real de WhatsApp.

### Workstream 7 — Inicio Do Piloto

1. Treinar `owner` e `manager`.
2. Treinar `receptionist` no escopo controlado.
3. Treinar `trainer` em `Assessments + task-lite`.
4. Operar com uma base pequena e horario acompanhado no primeiro dia.
5. Comecar observacao diaria usando:
   - `deploy/CONTROLLED_PILOT_RUNBOOK.md`
   - `deploy/CONTROLLED_PILOT_DAILY_CHECKLIST.md`
   - `deploy/CONTROLLED_PILOT_INCIDENT_TEMPLATE.md`

## Dependencias Criticas

- credenciais reais de Supabase
- Railway com billing ativo
- Vercel configurada
- SendGrid, se email entrar no piloto
- WhatsApp provider, se WhatsApp entrar no piloto

## Riscos Conhecidos

### Risco 1 — Ambiente sobe, mas worker nao processa

- sinal:
  - requests de recalc nao saem de `queued`
- mitigacao:
  - checar logs do worker
  - checar Redis
  - checar `ENABLE_SCHEDULER=true` no worker

### Risco 2 — Frontend sobe sem CORS correto

- sinal:
  - login/requests falham no browser
- mitigacao:
  - corrigir `FRONTEND_URL`
  - corrigir `CORS_ORIGINS`

### Risco 3 — Integracoes atrapalham o go-live

- mitigacao:
  - nao ligar Claude, Actuar ou leitura assistida no primeiro deploy
  - tratar email e WhatsApp como smoke test controlado, nao como dependencia de todo o fluxo

## Critério De Aceite Do Deploy

O deploy do piloto so e considerado pronto quando:

- `/health/ready` responde `200`
- login funciona
- importacao funciona com preview e commit
- requests de recalc manual de risco chegam a `completed`
- CRM read-only da recepcao esta coerente
- task-lite do trainer esta visivel e resolvivel
- WebSocket conecta e entrega evento em duas sessoes
- owner e manager conseguem operar o core sem erro bloqueante

## Rollback

1. Desligar o worker se o problema estiver em jobs.
2. Reverter backend no Railway.
3. Reverter frontend na Vercel.
4. Revalidar:
   - `GET /health/ready`
   - login
   - members
   - importacao
   - retencao

## Proximo Passo

Com este plano aprovado, o proximo passo operacional e executar os workstreams 1 a 4 em sequencia, sem abrir feature nova no meio do caminho.
