# 09.7 - Plano

## Resultado esperado

Um pacote V1 de agente WhatsApp-first no n8n, integrado ao backend existente sem assumir a Evolution API, com dois agentes separados: interno operacional e externo controlado.

## Etapas

1. Atualizar Spec Kit e GSD da fase existente.
2. Adicionar servico backend para chamar n8n e aplicar fallback legado sem duplicidade.
3. Adicionar endpoint backend protegido para reply do agente.
4. Reescrever `/n8n` para router WhatsApp, agente interno, agente externo e send reply via backend.
5. Manter tools operacionais como sub-workflows seguros.
6. Atualizar prompts para WhatsApp-first.
7. Atualizar schema SQL com eventos WhatsApp/dedupe.
8. Atualizar README, SECURITY, TEST_PLAN e env examples.
9. Atualizar drafts inativos no n8n usando SDK validado.
10. Validar backend, JSON, n8n e Spec Kit.

## Criterios de aceite

- Backend recebe Evolution e chama n8n com payload normalizado.
- Backend ignora grupo/fromMe/payload invalido/instancia desconhecida.
- Backend evita resposta duplicada com fallback legado.
- Router n8n retorna contrato esperado pelo backend.
- Agente interno e externo existem como fluxos separados.
- Externo nao envia para terceiros, nao faz broadcast e cria handoff para risco.
- Approval gateway, audit logger, task manager e RAG planejado existem.
- Drafts no n8n permanecem inativos.
