# Phase 4.1: Hardening P0 - Context

**Gathered:** 2026-03-25
**Status:** Executed

<domain>
## Phase Boundary

Esta fase existe para fechar os P0 do milestone antes de qualquer piloto controlado:

- aniversario importado com `birthday_label`
- recalc manual de risco sem thread daemon na API
- realtime/WebSocket funcionando com mais de um worker
- suites backend/frontend totalmente verdes

Ela nao abre feature nova de produto. O foco e previsibilidade operacional.
</domain>

<decisions>
## Implementation Decisions

- **D-01:** `birthday_label` segue como fallback aceito para dashboards e automacoes quando `birthdate` completo nao existir.
- **D-02:** `POST /members/recalculate-risk` deixa de executar trabalho em thread daemon; passa a criar uma solicitacao duravel.
- **D-03:** o worker processa a fila de recalc com lock distribuido e expoe um status consultavel por `request_id`.
- **D-04:** o contrato de WebSocket do cliente permanece; a mudanca e apenas de infraestrutura, usando Redis Pub/Sub por `gym_id` e `user_id`.
- **D-05:** a fase so fecha com backend e frontend verdes em teste, lint e build.
</decisions>
