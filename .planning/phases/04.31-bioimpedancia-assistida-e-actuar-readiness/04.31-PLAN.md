# Phase 4.31 Plan

## Goal

Fechar bioimpedancia assistida e Actuar readiness no piloto, com fluxo honesto e operacional mesmo quando a integracao nao estiver totalmente ativa.

## Requirements

- `BODY-01`
- `BODY-02`
- `BODY-03`
- `BODY-04`

## Workstreams

### 1. Backend capability contract

- revisar payloads de OCR parse e sync status
- garantir que os estados retornados diferenciem claramente:
  - OCR local
  - IA assistida
  - fallback manual
  - sync desabilitado
  - sync manual necessario
- garantir que campos nao suportados pelo Actuar estejam acessiveis para UI

### 2. Frontend readiness UX

- ajustar `MemberBodyCompositionTab` para refletir capability state real
- reduzir ambiguidade entre:
  - leitura assistida
  - OCR local
  - fallback manual
- tornar estado de Actuar desabilitado/parcial mais evidente
- revisar CTAs para nao sugerir automacao inexistente

### 3. Pilot-safe policy

- decidir o que o piloto chama de:
  - `leitura assistida`
  - `sync Actuar`
- se a integracao nao estiver ativa, rebaixar promessa sem quebrar o fluxo operacional
- revisar mensagens para owner, manager e trainer

### 4. Validation

- testes backend cobrindo estados de capability
- testes frontend cobrindo renderizacao dos estados principais
- smoke test no piloto para:
  - OCR local somente
  - IA desligada
  - Actuar desabilitado
  - fallback manual

## Deliverables

- backend com contrato mais explicito de readiness
- frontend sem promessa enganosa no fluxo de bioimpedancia
- documentacao curta do modo de operacao do piloto para bioimpedancia/Actuar

## Exit criteria

- o piloto deixa de parecer “integrado” quando ainda esta em modo manual assistido
- o usuario entende claramente o estado atual do fluxo
- trainer e manager conseguem operar o modulo sem improvisar o que o sistema realmente fez
