# Phase 7.0 Validation - Lead-to-member intelligence graph

Date: 2026-05-02
Verdict: PASS

## Scope checked

- Backend route: `GET /api/v1/members/{member_id}/intelligence-context`.
- Backend schema: `LeadToMemberIntelligenceContextOut` and nested semantic sections.
- Backend service: `get_member_intelligence_context` and `build_lead_to_member_intelligence_context`.
- Frontend contract: `LeadToMemberIntelligenceContext` and `memberService.getIntelligenceContext`.
- Operational consumers: Profile 360, Coach workspace, AI Inbox, Tasks, Retention, CRM and BI Foundation.

## Contract result

- Payload version remains `lead-member-context-v1`.
- Backend and frontend contracts still expose the same canonical sections:
  - `member`
  - `lead`
  - `consent`
  - `lifecycle`
  - `activity`
  - `assessment`
  - `operations`
  - `risk`
  - `signals`
  - `data_quality_flags`
- The endpoint is read-only and keeps the canonical namespace under `members/{member_id}`.

## Consumer result

- Profile 360 fetches `memberService.getIntelligenceContext(memberId)` and passes the payload to `AssessmentWorkspaceOverview`.
- `AssessmentWorkspaceOverview` renders both `CoachWorkspacePanel` and `MemberIntelligenceContextPanel` from that payload.
- AI Inbox uses `WorkExecutionView`, which fetches the same payload for the selected member and renders `MemberIntelligenceMiniCard`.
- Tasks, Retention and CRM drawers fetch the same payload and render `MemberIntelligenceMiniCard`.
- CRM only uses the canonical member context when `converted_member_id` exists; non-converted leads remain in the commercial flow.
- Remaining direct risk/lead fields in queues are source-row display fields, not UI-only recomputation of the canonical lead/member/consent/risk model.

## Tenant isolation

- The route depends on authenticated roles and passes `current_user.gym_id` into `get_member_intelligence_context`.
- The service starts with `get_member_or_404(db, member_id, gym_id=gym_id)`.
- Lead, check-in, assessment, body composition, task and risk alert queries include explicit `gym_id == gym_id` filters.
- `include_all_tenants(...)` is used only with the allowlisted `member_intelligence.explicit_gym_scope` reason and explicit tenant predicates.
- Consent status is loaded through `current_consent_status_map(db, member.id, gym_id=gym_id)`.
- Dashboard BI support resolves tenant context through the existing request `gym_id` context and adds explicit `gym_id` filters in cohort/follow-up helpers.

## Data quality

- Missing lead origin, consent, preferred shift, recent check-in, assessment and body composition stay explicit through `data_quality_flags`.
- BI Foundation keeps equivalent honest states for missing cohort, LTV, follow-up outcomes and fee base.
- No missing origin, consent, assessment or body composition value is inferred as truth in the canonical payload.
