## Entregue neste corte

- Schema semantico de Acquisition OS:
  - captura;
  - qualificacao;
  - resumo do lead;
  - resposta com lead, booking e score.
- Service `acquisition_service` com:
  - score deterministico de propensao;
  - motivos rastreaveis;
  - campos faltantes;
  - criacao de `Lead`;
  - criacao opcional de `LeadBooking` para aula experimental;
  - resumo semantico lido de notes estruturadas.
- Endpoints internos no namespace canonico de CRM:
  - `POST /api/v1/crm/acquisition/capture`;
  - `GET /api/v1/crm/acquisition/summary`;
  - `GET /api/v1/crm/leads/{lead_id}/acquisition-summary`.
- Endpoint publico protegido por token:
  - `POST /api/v1/public/acquisition/capture`.
- CRM frontend atualizado:
  - botao `Nova captura`;
  - campos de canal, campanha, objetivo, turno preferido, aula experimental, urgencia e consentimentos;
  - criacao usando endpoint de captura/qualificacao;
  - card `Acquisition OS` no detalhe do lead;
  - badges na lista com score, canal, turno e aula experimental;
  - busca incluindo canal, campanha, objetivo e turno.
- Contratos TypeScript para captura, qualificacao e resumo.
- T201 marcada como concluida em Spec Kit.

## Decisoes

- Sem migration nesta rodada: origem/campanha/qualificacao ficam como notes estruturadas para preservar compatibilidade.
- `LeadBooking` continua sendo a fonte de aula experimental quando existe horario real.
- O "chatbot qualificador" nasce como respostas estruturadas de qualificacao, nao como LLM conversacional.
- O endpoint publico usa o mesmo guardrail/token do booking publico; landing visual completa fica para fase posterior.
- Score e proxima acao sao deterministivos para manter explicabilidade e seguranca operacional.
