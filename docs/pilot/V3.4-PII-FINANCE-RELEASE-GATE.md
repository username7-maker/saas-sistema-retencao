# v3.4 Public PII / Finance Release Gate

Date: 2026-05-02
Scope: Acquisition OS, Compliance OS, Growth / Marketing OS, Finance Foundation, Operational Delinquency Ladder.

## Recommendation

**Split the release before pilot exposure.**

Public acquisition capture is acceptably gated by the existing public booking flag/token controls, but the complete v3.4 bundle is **not ready as one pilot publish**. Two security/privacy blockers need remediation or explicit security sign-off first:

1. Consent history responses expose `evidence_ref`, `notes`, `extra_data`, `gym_id`, and `member_id` through `MemberConsentRecordOut`, and `GET /api/v1/lgpd/member/{member_id}/consents` is also available to trainers. This does not satisfy the gate requirement that consent evidence/notes are not overexposed.
2. Delinquency finance endpoints and finance-domain Work Queue tasks are available to `receptionist`. This may be operationally intended by the delinquency spec, but it conflicts with this gate's owner/manager finance restriction unless `receptionist` is formally treated as a finance role for pilot.

Recommended rollout split:

- Allow a tightly controlled public acquisition pilot only when `PUBLIC_BOOKING_CONFIRM_ENABLED`, `PUBLIC_BOOKING_CONFIRM_TOKEN`, and `PUBLIC_DIAG_GYM_ID` are configured and the token is not exposed in URL/query strings.
- Hold Compliance OS consent history from broad staff pilot until the response model is minimized and role access is reviewed.
- Hold Finance Foundation and Delinquency Ladder from non-owner/manager pilot users until the receptionist policy is explicitly accepted or code is narrowed.
- Route SecurityEngineer review before publishing the full bundle. If no SecurityEngineer is available, CEO acceptance should explicitly cover the two blockers above.

## Tenant Isolation Review

Passes observed:

- Public acquisition resolves a configured public gym via `resolve_public_gym_id()` and passes that `gym_id` into `capture_acquisition_lead`.
- Acquisition creation validates optional `owner_id` through `ensure_optional_user_in_gym`; acquisition summary reads filter by `Lead.gym_id` and `deleted_at`.
- Consent service verifies the member with `member_id + gym_id` before listing or recording consent records.
- LGPD export/anonymization first verifies the member with `member_id + gym_id`; consent record export/redaction filters by `member_id + gym_id`.
- Growth lead/member/NPS queries filter by `gym_id` and active/deleted state.
- Finance service filters list/summary by `FinancialEntry.gym_id` and `deleted_at`; optional `member_id` and `lead_id` are validated by `tenant_guard`.
- Delinquency service derives items from `FinancialEntry.gym_id`, `deleted_at`, active members, and open/overdue receivables; created tasks carry `gym_id`.
- Work Queue list/detail/execute/outcome paths check `Task.gym_id == current_user.gym_id`; AI recommendations are fetched by `gym_id`.

Residual concern:

- Some LGPD related-table export queries use `member_id` without also filtering every related table by `gym_id`. The member is verified first and UUIDs are expected to be globally unique, so this is not a release blocker, but future hardening should keep tenant predicates explicit on every exported table.

## Gate Checks

### Public Acquisition Capture

Verdict: **Pass with audit caveat.**

- Uses the existing public booking enablement flag and shared-token guard.
- Production config enforces `PUBLIC_BOOKING_CONFIRM_TOKEN` when public booking/capture is enabled.
- Uses `resolve_public_gym_id()` and writes into that tenant, not an arbitrary tenant supplied by the caller.
- No public read/list endpoint for acquisition data was introduced.
- Response contains the newly created lead/booking/summary for the submitted payload; it does not read back existing tenant records.

Required before broader exposure:

- Add a dedicated public audit event.
- Confirm the shared token is delivered in a header or bearer token, never query string.
- Keep the endpoint disabled by default outside the pilot tenant.

### Consent Records and LGPD

Verdict: **Block for full pilot.**

Passes:

- `record_member_consent` only appends a new `MemberConsentRecord`; there is no update/delete consent endpoint.
- Current consent is derived from the latest record by type.
- LGPD export includes consent history metadata.
- LGPD anonymization redacts `evidence_ref`, `notes`, and `extra_data`.
- Consent write path logs `member_consent_recorded`.

Blocker:

- `MemberConsentRecordOut` exposes evidence, notes, and extra data in API responses. Combined with trainer read access to `GET /lgpd/member/{member_id}/consents`, this overexposes evidence/notes for the gate's privacy standard.

Required before release:

- Split response models: return current status and safe history metadata by default; reserve `evidence_ref`, `notes`, and raw `extra_data` for owner/manager-only audit export or a separate privileged endpoint.
- Decide whether trainer needs any consent history read. If yes, trainer should get current status only, not record evidence.

### Finance and Delinquency

Verdict: **Block or require explicit policy acceptance.**

Passes:

- Finance entry CRUD and finance summary are owner/manager only.
- Financial entries are tenant-scoped and soft-deleted.
- Optional member/lead links are tenant-validated before persistence.
- Finance mutations log audit events.
- Financial dashboard route is owner/manager only.
- Work Queue hides finance-domain tasks from trainer and salesperson.

Blocker / policy conflict:

- Delinquency endpoints and finance-domain Work Queue tasks allow `receptionist`.
- The delinquency spec asks reception/management to operate collections, but this gate asks finance/delinquency data to remain owner/manager surfaces and hidden from non-finance roles.

Required before release:

- Either narrow `FINANCE_OPERATION_ROLES` and Work Queue finance access to owner/manager only, or document that receptionist is a finance-operational pilot role and obtain CEO/Security acceptance.
- If receptionist remains allowed, minimize data shown in Work Queue to the operational message/task context and keep detailed amounts/history on owner/manager surfaces.

### Growth OS

Verdict: **Pass for authenticated staff pilot, with consent-warning limitation.**

- Authenticated only.
- Reads leads/members by tenant.
- Prepare action is human-triggered and does not send external messages automatically.
- Missing communication consent creates warnings instead of silent sending.
- Prepare action can create a task or lead note and logs `growth_opportunity_prepared`.

Limitation:

- Growth still surfaces opportunities even when consent is missing. This is acceptable only because action remains manual and warning-based in V1.

## Final Go / No-Go

**No-go for a single full v3.4 pilot publish.**

**Go for a split pilot only if:**

1. Public acquisition capture stays feature-flagged and token-protected.
2. Consent response overexposure is fixed before staff exposure.
3. Finance/delinquency receptionist access is either removed or explicitly approved as a finance role exception.
4. SecurityEngineer, or CEO if no SecurityEngineer is available, accepts the residual risk before exposing finance/delinquency in pilot.
